# Changelog

All notable changes to `pi-mmr` will be documented in this file.

The format follows the project [`docs/changelog-template.md`](docs/changelog-template.md).

## Unreleased

### Removed

- `mmr-subagents` / `mmr-core`: remove the hidden `cthulu` advisor easter egg and its R'lyehian rendering entirely. Deletes the `cthulu` Pi tool, its standalone subagent profile/prompt builder, the `cthulu` entries from the `mmr-subagents` tool provider/feature capabilities and the `mmr-core` tool-ownership registry, and the `cthulu` name from the `smart`, `smartGPT`, `rush`, `large`, and `deep` locked-mode tool allowlists. The `## The Sunken Rite` roleplay gate and `## Lingering style` blocks are dropped from every locked-mode prompt, and the `MMR_CTHULU_SUMMON_GATE` / `MMR_CTHULU_RITE_ANCHORS` exports and all `CTHULU_*` / `createCthuluTool` / `registerCthuluTool` / `buildCthuluWorkerSystemPrompt` package-root exports are removed. Prompt-assembly re-entrancy is preserved by replacing the gate-as-sentinel idempotency check with an exact structural match of the previously-injected MMR tail (shared tool guidance + shared coding guidance + mode posture + response style) across all known mode templates, so re-assembling an already-rewritten prompt still strips the prior MMR tail instead of duplicating it (including the cross-mode `deep`→`smart` Task base case). Regenerated `mmr-core-prompts`, `mmr-effective-surface`, and `mmr-subagent-surface` fixtures; removed `tests/mmr-subagents-cthulu.test.mjs` and `tests/mmr-subagents-rlyehian.test.mjs`; updated `tests/mmr-subagents-extension.test.mjs`, `tests/mmr-subagents-provider.test.mjs`, `tests/mmr-subagents-progress-rendering.test.mjs`, and `tests/mmr-tool-execution-mode.test.mjs`.

### Added

- `mmr-subagents`: lower friction for custom Markdown subagents with sensible frontmatter fallbacks plus a user-facing notice. A definition that declares no `tools` (or `allowed-tools`/`allowedTools`) field now defaults to a fixed standard toolset — `read, bash, edit, write, find, grep, web_search, read_web_page` (exported as `MMR_CUSTOM_SUBAGENT_DEFAULT_TOOLS`) — each still intersected with the registered/active tools. A fixed constant is used rather than "all registered tools" so the parent and the spawned child resolve the same set and the worker never fails activation on a tool mismatch; recursive/advisory subagents, toolbox, and MCP tools are excluded by construction. An explicitly empty list (`tools:` or `tools: []`) still runs with no tools so an author can deliberately build a prompt-only subagent. The loader also parses a provider-neutral thinking/effort level from `thinkingLevel` / `thinking` / `effort` (canonical Pi levels `off`/`minimal`/`low`/`medium`/`high`/`xhigh`, case-insensitive, reusing `mmr-core`'s `isThinkingLevel`; no vendor aliases) and wires it onto the runtime profile so the worker honors it, and records `modelDeclared` so an omitted `model` key inherits the parent model. When the subagent relied on a fallback for `model`, thinking level, or `tools` (including the explicit no-tools case), a single advisory listing each fallback is surfaced **only** in the rendered tool result and as `details.fallbackNotice` (built by the exported `buildMmrCustomSubagentFallbackNotice`); it is deliberately never injected into the model-consumed `content`, so the parent model sees only the worker's answer while the human author sees the recommendation to pin those fields. `MmrCustomSubagentDefinition` gains `modelDeclared` and `thinkingLevel`; `toolsDeclared` and the exported `MMR_CUSTOM_SUBAGENT_TOOL_KEYS` are retained. Covered by `tests/mmr-subagents-custom-loader.test.mjs` and `tests/mmr-subagents-custom-runtime.test.mjs`.
- `mmr-subagents`: support user-authored Markdown subagents as `sa__*` Pi tools executed through the shared subagent runner with the Markdown body as the worker system prompt (see the config-driven setup/import revision under `### Changed` for the final shipped roots/enablement model). Custom tools get model-visible descriptions/guidelines from their frontmatter, runtime profiles/prompt builders for child activation, `model: inherit` support via the existing subagent model-override channel, and a tool allowlist filtered through the parent-active registered tool surface so denied tools do not leak into custom workers. Discovery is bounded to Claude Code-style local-agent scale (100 definitions / 1000 Markdown files by default), refuses symlink roots/entries, and rechecks realpath containment under the configured root before reading. Covered by `tests/mmr-subagents-custom-runtime.test.mjs` and `tests/mmr-subagents-custom-loader.test.mjs`.
- `mmr-debug`: add a developer-only, opt-in live capture extension that records the ground-truth provider request/response surface for debugging model behavior. It hooks `before_provider_request` (the assembled system prompt + advertised tool names Pi actually serialized — authoritative because payload-level system-prompt rewrites are not reflected by `ctx.getSystemPrompt()`), `after_provider_response` (HTTP status + headers; the hook exposes no body), and assistant `message_end` (the model's output), appending one JSON Lines record per event tagged with `turn`, a monotonic `seq`, an ISO timestamp, and the session id so a multi-turn bug can be diffed turn-by-turn. It is fully inert unless `MMR_DEBUG_CAPTURE_FILE` is set (no env → no hooks registered → zero cost); `MMR_DEBUG_CAPTURE_FULL=1` additionally dumps the entire raw request payload. Records are written with mode `0600` and the extension never modifies the payload (read-only). It is intentionally NOT registered in `package.json` `pi.extensions` (never auto-loaded for users) and is excluded from the npm package via `.npmignore`; load it explicitly with `pi -e "$PWD/src/extensions/mmr-debug/index.ts"`. The recommended gitignored capture path `.pi/mmr-debug/` plus `*.mmr-capture.jsonl` are added to `.gitignore` because the output contains full prompt/session text. The pure, Pi-free extractors (`extractSystemPrompt`, `extractToolNames`, `extractMessageSummary`, `stringifyContent`) handle the Anthropic `system` string/text-block, OpenAI `instructions`, and OpenAI Responses `input[]` role-message shapes and are covered by `tests/mmr-debug-capture.test.mjs`. Not re-exported from the package root, since the extension is not part of the shipped public surface.

### Changed

- `mmr-subagents` / `mmr-core`: make custom Markdown subagents config-driven instead of auto-loaded from `.claude/agents`. Runtime discovery roots are now Pi-owned (`<cwd>/.pi/subagents` and `~/.pi/agent/subagents`); `.claude/agents` is scanned only as an import-candidate source by the setup/import flow and is never auto-registered. Config is the enablement boundary: a Markdown file on disk is a *candidate* and is not registered or model-visible until an enabled `mmrSubagents.custom.agents.<id>` record (global or project settings; project overrides by id) references it. Each record carries `enabled`, `source { root, file }`, `toolName`, `modes` (`"allLocked"` or a locked-mode list), optional global-only `projects` (`"all"` or exact cwd paths), `model`, optional `thinkingLevel`, and `tools`; the record's fields win over the Markdown frontmatter. Per-mode exposure is implemented through a new `mmr-core` mode-extra-tool provider hook (`registerMmrModeExtraToolProvider` / `resolveMmrModeExtraTools`): an enabled subagent registers as a tool but only enters a locked mode's active set when that mode is in its `modes` scope, merged through the same additive, fail-closed path as `lockedModeExtraTools`, and never appears in Free mode. A `/mmr-config` → "subagent (setup/import custom)" wizard scans Pi-owned and legacy Claude candidates, recommends model/thinking/tools (least-privilege read-only by default — a missing source `tools` list never means "all Pi tools"), maps Claude tool aliases, blocks recursive/advisory/MCP/mutation tools, surfaces unknown tools and unavailable declared models, asks for modes and project scope, copies external Markdown into a Pi-owned root, and writes the config record. When legacy `.claude/agents` candidates exist but nothing is enabled, a one-time-per-session migration notice points the user at the setup/import flow (no compatibility switch restores the old auto-loading). Safety hardening: the reserved `sa__*` namespace is excluded from the user-controlled `mmrCore.lockedModeExtraTools` path (`excludeReservedSubagentNames` / `MMR_RESERVED_SUBAGENT_TOOL_PREFIX`) so a custom subagent can only ever enter a mode through the scope-aware provider; enabled record source files are read with `O_NOFOLLOW` + realpath containment under their Pi-owned root (a symlinked source escaping the root is refused); global `projects` entries must be absolute cwd paths (relative entries are dropped with a warning); `toolName` must match `sa__<slug>` (lowercase/digits/underscore, length-capped); reserved record ids (`__proto__`/`prototype`/`constructor`) are skipped on load and rejected by the writer; and the setup wizard confirms before overwriting an existing Markdown file or config record, flags unknown imported tools against the live tool inventory, and tells the user a restart is required (custom subagents load once at activation). New `custom-config.ts` (load/merge/scope/write records), `custom-import.ts` (pure tool-mapping/diagnostics import planner), and `config-flow.ts` (wizard + pure helpers); `MMR_CUSTOM_SUBAGENT_RECOMMENDED_READONLY_TOOLS`, `MMR_RESERVED_SUBAGENT_TOOL_PREFIX`/`excludeReservedSubagentNames`, and the `MmrModeExtraToolProvider` type are added. Covered by `tests/mmr-subagents-custom-config.test.mjs`, `tests/mmr-subagents-custom-import.test.mjs`, `tests/mmr-subagents-config-flow.test.mjs`, `tests/mmr-core-mode-extra-tools.test.mjs`, expanded `tests/mmr-subagents-custom-runtime.test.mjs`, and a new reserved-name case in `tests/mmr-core-extra-tools.test.mjs`.
- `mmr-toolbox`: strengthen `task_list` reliability during long work. The tool prompt now gives stricter usage/completion rules, successful writes remind the agent to keep the list current, all-completed 3+ item lists without an explicit verification/check step produce a verification nudge, and the per-call `context` hook injects a bounded stale-update reminder after repeated model calls without a `task_list` write. Covered by `tests/mmr-toolbox-todo-list-tool.test.mjs` and `tests/mmr-toolbox-todo-list-session-scope.test.mjs`.
- `mmr-core`: refactor locked-mode prompt assembly around a Pi-native base prompt plus ordered prompt fragments and per-mode recipes. The new `prompt-registry.ts` maps Pi-owned passthrough sections (`Available tools`, `Guidelines`, Pi documentation, preserved tail) separately from MMR-owned fragments (tool lead-in, built-in guidance, shared guidance, mode posture, and response style), exposes the registry through the package root, and keeps rendered locked-mode prompts byte-identical to the previous snapshots. Covered by `tests/mmr-core-prompt-registry.test.mjs` and expanded prompt-assembly invariants.
- `mmr-core`: split the monolithic shared coding-guidance block into named prompt fragments (`autonomy`, `discovery-discipline`, `pragmatism`, `verification`, `careful-actions`, `diagrams`, `file-links`, `collaboration`) so each mode recipe can include only the sections it needs. `prompt-modules.ts` now exposes `SHARED_CODING_GUIDANCE_FRAGMENTS` (with `SHARED_CODING_GUIDANCE` derived from it for the byte-reference), the registry's default fragment sequence and `MmrPromptBlockKind` vocabulary enumerate the granular fragments, and `assembleActiveSurface` renders rendered block kinds equal to the active recipe's fragment list. The default recipe keeps every fragment in the original order, so `smart`, `smartGPT`, `large`, and `deep` prompts stay byte-identical to the previous snapshots; only `rush` changes, dropping the `diagrams` fragment for token economy via the new `MMR_RUSH_PROMPT_FRAGMENT_SEQUENCE`. The root-exported `MmrPromptBlockKind` type drops `shared-coding-guidance` in favor of the granular kinds. Covered by expanded prompt registry/assembly tests and regenerated `rush` fixtures.

### Fixed

- `mmr-subagents`: serialize an empty worker tool ceiling as an explicit `--tools ""` instead of dropping the flag. `buildMmrWorkerArgs` previously omitted `--tools` whenever the tools array was empty, but the child's `--tools` parser already treats `--tools ""` as "the runner explicitly asked for no tools". The drop meant a spawned worker whose parent-resolved tool set reduced to empty (e.g. an explicit-list custom subagent whose listed tools are all inactive in the parent) would fall back to its own profile-resolved set at child activation instead of running with no tools — a least-privilege gap. The flag is now emitted whenever the caller passes a tools array (including empty); callers that intend the child to self-resolve (finder/oracle) continue to omit `tools` entirely. Covered by a new `buildMmrWorkerArgs` case in `tests/mmr-subagents-task.test.mjs`.

## 0.2.0 — 2026-06-05

### Added

- `mmr-subagents`: capture the parent turn's structured `systemPromptOptions` (Pi's `BuildSystemPromptOptions`) alongside the parent system-prompt string used to build mode-derived `Task` workers, so worker diagnostics can read the parent surface. `registerTaskParentPromptCapture` now records both atomically — the options are only updated when a non-empty prompt string is seen, so they never desync from a stale earlier turn — and the new `getTaskParentSystemPromptOptions()` exposes a minimal validated view (`selectedTools` filtered to strings with an empty selection preserved as distinct from not-supplied, non-empty `customPrompt`). Metadata only: it is never used to rebuild a prompt, and worker prompt assembly and routing are unchanged. Covered by `tests/mmr-subagents-task-parent-prompt.test.mjs`.
- `mmr-subagents`: add async background subagent task tools (`start_task`, `task_poll`, `task_wait`, `task_cancel`) so the parent agent can launch a bounded worker, keep working, and poll/wait/cancel it — the blocking `Task` tool is unchanged. The four tools share `Task`'s validation/routing path via the new exported `prepareTaskRun` plus `buildTaskProgressResult`/`buildTaskFinalResult`/`buildTaskRunnerThrowResult`/`resolveTaskRunner` helpers (zero behavior change to blocking `Task`). Background runs are tracked in a new in-memory, session-scoped registry (`async-task-registry.ts`, a `globalThis` singleton with a shape-compatibility guard like `mmr-core/runtime.ts`): the registry owns each worker's `AbortController` (a background run is never bound to the per-call tool `signal`), derives liveness from heartbeat recency + a max-runtime watchdog (`healthy`/`stalled`/`dead`) instead of trusting a static `running` flag, enforces a per-session concurrency cap, dedupes retried starts by originating tool-call id, guards against late progress/finalize writes, prunes terminal records on a dual TTL (longer before first observation, shorter after), and aborts active workers + clears records on `session_shutdown`. `task_poll` returns a single task's progress/final result or, with no id, a partitioned board (active/stalled/finished); `task_wait` blocks up to a bounded `timeout_ms` without cancelling the worker on timeout; `task_cancel` is idempotent and synthesizes an Amp-style cancellation summary from the worker trail. State is in-memory and process-local only (no disk, lost on restart), and the four tools are denied inside the `task-subagent` worker surface so a worker cannot spawn background workers. They ship behind a distinct `mmr-subagents.async-tasks` feature gate (added to the `smart`, `smartGPT`, `rush`, `large`, and `deep` modes and reserved in `mmr-core`'s feature-gate list), are credited to `mmr-subagents` in the tool catalog/provider, and re-export from the package root. At-most-once completion push (a one-time poke when a background task finishes) is on by default via a two-layer gate: a session ceiling, on by default and force-disabled by the `MMR_SUBAGENTS_ASYNC_PUSH` environment gate, plus a per-task `start_task({ notify: false })` opt-out (the `notify` flag is stripped before the shared `Task` validator and never affects worker prompt/routing). When both allow it, the registry delivers exactly one `pi.sendMessage({ deliverAs: "nextTurn", triggerTurn: true })`, which wakes the session only when idle and otherwise rides the next turn, and a per-session push budget (`DEFAULT_ASYNC_TASK_MAX_PUSHES_PER_SESSION`, recorded as a `suppressed` push state once exhausted and cleared on `session_shutdown`) caps how many times a session can self-wake. The registry runs an active per-task max-runtime watchdog (a runaway worker is aborted even with no further polling, with prune-time enforcement as a backstop); `session_shutdown` invalidates any in-flight worker callbacks, clears the tool-call idempotency index, and disables completion push so a late finalize cannot mutate state or notify after teardown; and a background spawn failure finalizes with the same `spawn-error` status/shaping as blocking `Task`. The registry exposes a full internal snapshot to the in-package tools but the package root publishes only a lean public `MmrAsyncTaskSnapshot` (identity/status/timing plus `promptChars`/`hasFinalResult`) built via the exported `toPublicAsyncTaskSnapshot` projector, so the prompt text, cwd, worker tools, latest-progress payload, and full worker result (which can include stderr/args/trail) are not part of the stable public surface. Covered by `tests/mmr-subagents-async-task-registry.test.mjs` and `tests/mmr-subagents-async-task-tools.test.mjs`, with updated provider/extension/profile/resolve assertions.
- `mmr-subagents` / `mmr-core`: add a session-scoped subagent worker-model fallback that prompts before switching routes after repeated worker-model errors (mirrors the locked-mode quota fallback, scoped to spawned worker routes). When a `finder`, `oracle`, `librarian`, `Task`, or `cthulu` worker fails with a retryable worker-model error twice for the same session/profile/route, an interactive parent (`ctx.hasUI`) is prompted to pick a fallback from the profile fallback chain / locally-authenticated registry; each candidate is labelled with provider/model and billing route type (`subscription` / `API key` / `unknown billing`). The fallback is never auto-applied, so a subscription-backed route is never silently swapped for an API-key-billed one. An accepted choice re-runs the worker once and is held in-process for that session + subagent scope only (never written to settings). The selection is forwarded to the spawned child Pi through a runner-owned env channel (`PI_MMR_SUBAGENT_MODEL_PREFERENCES`, parsed/applied by `mmr-core` child activation with precedence over settings and profile defaults) so parent spawn and child activation resolve the same route and `--model` does not mismatch; the runner always passes a scrubbed copy of the child environment so a nested worker spawn (e.g. a fallback-routed `Task` child that calls `finder`) never inherits a stale override. Consecutive-failure counts reset on any non-retryable outcome (including success), and all in-process fallback state is cleared on a new/forked session (never inside a worker, never on resume) so it cannot leak across sessions. A chosen fallback is persisted for the scope only when the retry actually adopts that route, so a non-adopted override can never silently block future prompts. Prompt candidates mark billed (`API key`) and unverified routes with a leading warning marker while subscription routes stay unmarked, without changing candidate ordering. Only `worker-error` counts as retryable; `spawn-error`, `activation-error`, `aborted`, and empty/no-start runs never trigger a prompt. New `mmr-subagents/fallback.ts` (classification, billing-labelled candidate building, in-process state, prompt, and the `runMmrWorkerWithModelFallback` orchestrator), `mmr-core/subagent-model-override-env.ts` (env serialize/parse), and the exported `parseMmrModelPreferenceValue` settings helper; covered by `tests/mmr-subagents-fallback.test.mjs`, `tests/mmr-subagents-runner-fallback-env.test.mjs`, `tests/mmr-core-subagent-model-override-env.test.mjs`, and a new env-precedence case in `tests/mmr-core-subagent-activation.test.mjs`.
- `mmr-core`: add the `mmrCore.lockedModeExtraTools` setting so users can keep their own extension tools, third-party tools, or MCP tools callable while a locked mode is active, without releasing to Free mode. The setting maps an `all` bucket (every locked mode) and per-locked-mode keys (`smart`, `smartGPT`, `rush`, `large`, `deep`) to arrays of exact Pi tool names; `free` and unknown keys are ignored with a warning, names are trimmed/deduped, and global/project settings merge additively per key. Extras resolve by exact identity against Pi's live tool inventory (no aliases), are merged into the mode's active set after the base allowlist, and are credited to a `user-allowlist` owner in diagnostics when they resolve through plain identity. The fail-closed contract is preserved: extras never satisfy the zero-active-tools activation abort (only a mode's own tools can), and a missing extra is a non-fatal no-op surfaced as `missing`. Extras apply to the parent session only, never to subagent workers. New `selectExtraToolNames`/`mergeToolResolutions`/`relabelExtraOwners` helpers in `extra-tools.ts`; covered by `tests/mmr-core-extra-tools.test.mjs` and new `lockedModeExtraTools` cases in `tests/mmr-core-settings.test.mjs`.
- Repository hygiene: add a CI workflow, security policy, contributor guide, code-of-conduct, PR template, issue forms, CODEOWNERS, and `.editorconfig` so new contributors get clear checks and safe public-reporting paths. The contributor guide and PR template now also document the issue → branch → PR → checks loop, linked issues, labels, and Conventional Commit messages.
- `mmr-github`: add a new extension that ships seven read-only GitHub repository tools (`read_github`, `list_directory_github`, `glob_github`, `search_github`, `commit_search`, `diff_github`, `list_repositories`) backed by a shared read-only client (`createGithubClient`) and a repository-reference parser (`parseGithubRepository`) that accepts `owner/repo` or `https://github.com/owner/repo` and rejects search/organization/profile pages. Network access is off by default behind `MMR_GITHUB_ENABLE`; the token is environment-only (`MMR_GITHUB_TOKEN` / `GITHUB_TOKEN`) and settings-file tokens are ignored with a warning. The client builds bounded GET requests (timeout + byte caps via the shared `mmr-web` HTTP helpers, `X-GitHub-Api-Version`, bearer auth when a token is present) and maps 401/403-rate-limit/404/422 responses to clear errors. `read_github` fetches the whole file (bounded by GitHub's 1 MB contents-API inline ceiling via a dedicated ~1.5 MB read budget), applies `read_range` first, and then gates the resulting slice at 128 KiB, so a large file stays readable through a sufficiently small `read_range` and an oversized slice returns a clear "retry with a smaller read_range" error with the file's total line count; directory listings are gated the same way and `list_directory_github` accepts a bounding `limit`. The extension records its entrypoint source path so consumers can confirm tool ownership by source, exposes `mmr-github` feature-gate and tool providers, and re-exports its public surface from the package root (`createMmrGithubExtension`, `loadMmrGithubSettings`, `createMmrGithubToolProvider`, `createMmrGithubFeatureGateProvider`, `MMR_GITHUB_TOOL_NAMES`, `hasMmrGithubOwnedTools`, `createGithubClient`, `parseGithubRepository`, `registerMmrGithubTools`, `MMR_GITHUB_PROMPT_GUIDELINES`, and the `MMR_GITHUB_*` identifiers). Covered by `tests/mmr-github-config.test.mjs`, `tests/mmr-github-client.test.mjs`, `tests/mmr-github-tools.test.mjs`, `tests/mmr-github-provider.test.mjs`, and `tests/mmr-github-package.test.mjs`.
- `mmr-github`: tighten the read-only tool behavior for closer GitHub-tooling parity. `search_github` caps each context fragment at 2048 characters. `commit_search` searches commit messages with `author-date` qualifiers (sorted newest-first) when a `query` is given without a `path`, lists commits via the REST commits endpoint when a `path` is given or no query is supplied, and filters a `query`+`path` listing client-side by message/author/email; its default limit is 50 and long messages are truncated to 1024 characters. `diff_github` now defaults to file-level stats only and gates unified hunks behind an `includePatches` flag (each patch truncated to ~4 KB). `glob_github` renames its glob parameter to `filePattern`, supports `{a,b}` alternation and `[...]` character classes in addition to `*`/`**`/`?`, and fails fast when the repository tree is too large to list recursively. `list_repositories` switches to `pattern`/`organization`/`language` filters, prioritizes repositories the configured token can access (`/user/repos`) and supplements with public repository search, and falls back to public search when no token is available. `read_github` numbers lines as `N: line`, and a directory path supports `read_range` paging with `[... omitted N entries ...]` markers; `list_directory_github`'s default entry limit is 100. Covered by expanded `tests/mmr-github-tools.test.mjs` and `tests/mmr-github-client.test.mjs`.
- `mmr-subagents` / `mmr-github`: rewire the `librarian` worker from the public-web research surface to the read-only GitHub repository provider for full repository-research coverage (file reads, directory listings, glob lookups, code search, commit history, and ref diffs). The `librarian` profile's tool allowlist is now the seven `mmr-github` tools; `LIBRARIAN_WORKER_TOOLS` still derives from the profile. Librarian gating moves from "mmr-web web tools active" to "mmr-github tools registered and source-owned" (`isLibrarianGithubToolPrerequisiteRegistered`, replacing the previous web-prerequisite helpers): because the GitHub tools are registered globally but are not part of any user-facing mode's active set, the parent gate checks registration + source ownership rather than parent-active state, and the child worker activates them by name through its profile allowlist. Child activation fail-closes when the GitHub tools are not source-owned by `mmr-github`. The librarian worker system prompt and tool description now describe the GitHub repository provider and GitHub `blob/<revision>` link construction. The gating reason is now `librarian: requires mmr-github read-only GitHub tools (set MMR_GITHUB_ENABLE=true).`. `mmr-github` is registered in `package.json` after `mmr-core` and before `mmr-subagents`. Covered by updated `tests/mmr-subagents-librarian.test.mjs`, `tests/mmr-core-subagent-activation.test.mjs`, `tests/mmr-core-subagent-profiles.test.mjs`, `tests/mmr-core-subagent-resolve.test.mjs`, `tests/mmr-subagents-prompts.test.mjs`, `tests/mmr-subagents-provider.test.mjs`, `tests/mmr-subagents-extension.test.mjs`, and a regenerated `librarian-github` subagent-surface fixture.

### Changed

- Widen the Pi host peer range to admit the `0.78.x` line: set `package.json` `peerDependencies` for `@earendil-works/pi-coding-agent`, `@earendil-works/pi-agent-core`, and `@earendil-works/pi-tui` to `>=0.77.0 <0.79.0` (and the matching `package-lock.json` root spec). The previous `^0.77.0` resolves to `>=0.77.0 <0.78.0`, which excluded the current Pi `0.78.x` release line, so loading `pi-mmr` under Pi `0.78.x` (and the `pi -e "$PWD" --list-models` smoke check) reported a peer-version mismatch. `pi-mmr` uses no `0.78`-only API, so the source surface is unchanged; the new `ctx.mode` / `ctx.getSystemPromptOptions()` context additions remain optional and are not yet adopted. Local development and CI keep resolving the pinned `0.77.0` peers (which still satisfy the widened range) until the dev install is refreshed to `0.78.x`.
- `mmr-subagents`: render the async-task completion push as a compact background-task status row instead of dumping its raw `<task-notification>` block into the transcript. The push message keeps its model-facing XML `content` (the agent still consumes the same notification next turn), but a registered message renderer for the `mmr-subagents.async-task-completion` custom type now draws a single styled row from the message's structured `details` — the task's status (completed/failed/cancelled with the matching color), its description, and a `task_poll({task_id:"..."})` hint. The notifier's `details` now also carries the task `description` so the row needs no XML parsing, the renderer falls back to the host's default box on malformed/legacy details, and registration is feature-detected so headless/JSON hosts are unaffected. Covered by new `renderAsyncTaskCompletionMessage` cases in `tests/mmr-subagents-progress-rendering.test.mjs`.
- `mmr-subagents`: extend `start_task` so background runs can launch `Task` (default), `finder`, or `librarian` via an `agent` selector plus agent-specific `params`; `oracle` stays blocking-only and is rejected as a background agent. `task_poll`, `task_wait`, and `task_cancel` continue to manage all background workers through the same session-scoped task ids, and background task tools now render through a collapsed subagent-style surface that shows only invocation, background-running/completed state, and final output (no streamed worker trail). Legacy `start_task({prompt, description})` remains a `Task` shortcut. Covered by `tests/mmr-subagents-async-task-tools.test.mjs` and `tests/mmr-subagents-progress-rendering.test.mjs`.
- `mmr-core`: keep Smart's Anthropic output budget at 32k when toggling from the default high-effort shape to the xhigh-effort shape, so both Smart reasoning presets share the same admission-sized output reservation.

### Fixed

### Added

Storage lives beside the existing process-global runtime singleton in `src/extensions/mmr-core/runtime.ts`. The new public APIs are `getMmrSessionIdentity()` (live, deep-frozen), `getMmrSessionIdentitySnapshot()` (mutable deep clone), and `onMmrSessionIdentityChanged(pi, handler)` (per-handler deep clone subscription). The raw setter `setMmrSessionIdentity(...)` is intentionally **not** re-exported from the package root: identity should flow from Pi's session lifecycle in `mmr-core`. The runtime reports a `{ changed }` flag so `MMR_EVENT_SESSION_IDENTITY_CHANGED` emits only when the identity actually changes (idempotent re-sets and `undefined`→`undefined` clears do not re-emit); `observedAt` is intentionally excluded from identity equality so re-capturing the same session does not spuriously re-emit. `mmr-core`'s `session_start` hook captures `cwd`, `sessionId`, and `sessionName` from `ExtensionContext`/`ReadonlySessionManager` and publishes the event. The bus payload is the deep-frozen live identity; raw `pi.events.on(...)` subscribers must not mutate it, mirroring `MMR_EVENT_STATE_CHANGED` semantics.
- `mmr-core` introduces a `smartGPT` mode: a smart-style balanced posture routed to `gpt-5.5` with OpenAI Responses `xhigh` reasoning. Tool set matches smart (including the interim `task_list` registration), `promptRoute` is `default`, and the mode joins the `alt+m` picker rotation and the `ctrl+space` cycle. Use it for now; we can remove it later when smart-on-Claude is the only Smart variant we want exposed.

### Changed

- `mmr-core` now exposes the `task_list` tool in every enforced mode (smart, rush, large, deep) instead of only rush. This is an interim measure: when the future `mmr-subagents` task agent ships, modes that adopt it will drop `task_list` from their public tool list in favor of the agent. Free mode is unchanged (no tool enforcement).
- Mode tool sets refreshed: normal mode tool sets omit `code_review` because review orchestration is workflow-internal and out of scope, and no longer include diagnostics in rush. Docs now classify public mode tools separately from workflow-internal, subagent-only, server/internal, and omitted tool names; review workflow orchestration is user-owned/out of scope for pi-mmr for now.
- Free mode now behaves like Pi as if the `pi-mmr` package were not installed: it drops tools registered by MMR package extensions (`apply_patch` and `task_list` from `mmr-toolbox`; `web_search`, `read_web_page` from `mmr-web`) from the baseline before handing control back to Pi. Pi rebuilds its system prompt and provider payload `tools` from the active tool set, so those tools' descriptions and schemas no longer reach the model in Free mode. Tools registered by unrelated extensions are preserved. A new shared registry in `src/extensions/mmr-core/owned-tools.ts` tracks MMR-owned concrete tool names; sibling extensions register their tools there at load time.
- Free mode ownership filtering is now source-aware. Each MMR package extension entrypoint records its absolute filesystem path via `registerMmrOwnedExtensionPath(...)`, and Free mode cross-references the active `ToolInfo.sourceInfo.path` Pi reports for each tool. If a third-party extension re-registers a name MMR also owns (e.g. another package's `apply_patch`), its registration is preserved because its source path is not in the MMR-owned set. When Pi does not provide a source path, Free mode falls back to the existing name-based filtering so MMR-owned tools are still removed.

### Security

- `mmr-web` URL policy no longer accepts trailing-dot loopback hostnames (e.g. `http://localhost./`, `http://service.local./`) and no longer accepts IPv4-mapped IPv6 addresses in canonical hex form (`::ffff:7f00:1`, equivalent to `127.0.0.1`). The IPv6 range check now operates on the parsed 16-byte representation rather than the URL parser's normalized string, so it also catches NAT64 (`64:ff9b::/96`) and 6to4 (`2002::/16`) addresses that embed private IPv4, plus the multicast (`ff00::/8`) and unspecified (`::`) ranges.

### Documentation

- `src/extensions/mmr-toolbox/README.md` now documents the **multi-surface design pattern** future toolbox tools must follow: full guidance (grammar, context/matching rules, additional rules, reliability tips, worked examples, and inline departure notes) belongs in the tool `description`; short high-signal cues belong in `promptGuidelines`; the one-line `promptSnippet` belongs in the tool list; and human/implementer context belongs in the README. Each surface is described with its audience, expected density, and the kind of content it carries. `apply_patch` is documented as the reference implementation; the section also covers behavioral departures and how description-level surface assertions are tested. `docs/reference-architecture.md` cross-links to the new section from the `mmr-toolbox` module note.
- IDE diagnostics are not planned as a local `mmr-toolbox` tool. Pi is CLI-only with no native IDE/LSP bridge, so diagnostics belong to user-configured MCP/IDE integrations under their own tool names; `mmr-core` prompt templates direct the model to verify via bash typecheck/lint/build.

### Changed

- `apply_patch` tool description now covers the full schema description (formal grammar block; context rules with 3-line default and 5-10-line guidance for repetitive/large files; additional rules including conflict-marker length and the linter/formatter caveat; reliability tips for repeated blocks, partial reads, indentation, unanchored insert-only hunks, whitespace drift, and CRLF; and worked examples for add, simple update, scoped update, repetitive file, multi-`@@` narrowing, end-of-file anchor, jj conflict markers, delete, and move/rename). Two pi-mmr-specific behaviors (ambiguous matches are rejected rather than first-match-wins, and path safety extends to the workspace plus same-repo sibling worktrees) are documented inline so deep-mode models see them through the schema instead of having to consult external docs. The schema itself is unchanged (`{ patchText: string }`). `promptGuidelines` additionally surfaces four short cues in the generated system prompt: prefer `apply_patch` for single-file/patch-style changes over Python/shell rewrites; use the patch envelope and operation headers; include 3+ context lines (5-10 or an `@@` anchor for repetitive files); avoid unanchored insert-only hunks.
- `apply_patch` visible-text contract: `content[0].text` now begins with a single `Applied patch: …` status line followed by a blank line, then the structured diff body. Single-file patches use `Applied patch: <path> (+a/-d)`; multi-file patches use `Applied patch: N files` followed by labeled per-file sections (`<path> (+a/-d)` headers). API surfaces that hide the structured `details` field now show the LLM an unambiguous success marker instead of a bare diff body, which previous wording made easy to misread as a partial/failed tool result. `details` is unchanged. The interactive TUI `renderResult` path continues to render from `details` (Pi's `renderDiff` + numbered diff format, plus per-file path headers in multi-file patches), so the visible status line is naturally suppressed there — the diff already renders visually. Downstream consumers parsing `content[0].text` as a pure diff must strip the status line and the blank line that follow it (a trivial two-line skip).
- Locked MMR modes now own the interactive footer instead of adding an extra extension-status row. The footer keeps Pi-native working-directory, git branch, cumulative token/cost, subscription, and context-usage presentation, but replaces the right side with `<model> • <mode>` (for example `opus-4.8 • smart`). `/mode free` restores Pi's native footer.
- The MMR footer now displays the active mode's context window, so locked modes show `smart` 300k, `rush` 400k, `large` 1M, and `deep` 400k even when a provider registration reports a different model window.
- `apply_patch` path safety now extends from `ctx.cwd` only to `ctx.cwd` plus sibling worktrees of the same git repository (discovered via `git -C <cwd> worktree list --porcelain`, with each entry canonicalized and verified to share the current repo's git common dir). Unrelated sibling directories, `..` escapes, and symlink escapes from every allowed root remain rejected. Boundary errors now include the current workspace, the discovered worktree roots, and the rejected target.
- `apply_patch` now returns a structured result shape: `content[0].text` is a display diff body (context lines plus `-` removed and `+` added lines, without unified-diff metadata like `---`, `+++`, or `@@`), so visible tool output shows what changed even when a UI/API surface does not render `details`. In the interactive TUI, `apply_patch` now renders results with Pi's built-in `renderDiff` using the same numbered diff format that the built-in `edit` tool produces (` 73 context`, `-75 removed`, `+75 added`), so apply_patch results match `edit` output: same colors, same intra-line highlighting, same spacing. Multi-file patches prepend a labeled `path (+a/-d)` header before each file's diff. `details` carries `{ summary, files }` where `summary` is the compact per-file summary and each `files[]` entry includes `type` (`add` | `update` | `move` | `delete`), workspace-relative `path`, optional `oldPath` for moves, `file://` `uri`, `additions`, `deletions`, and a unified `diff` for parser/UI consumers. Replaces the legacy `Applied patch to N files:` text and the `{ files: [{ path, kind, movedTo? }] }` return type.
- `apply_patch` per-file `diff` now emits the standard `\ No newline at end of file` marker when either side of the change lacks a trailing newline, so a trailing-newline-only delta is reflected in `additions`/`deletions` and `git apply` can round-trip the diff. The line-LCS used to compute the diff is now bounded by a 4M-cell budget; whole-file diffs above that budget fall back to a clearly-marked `[unified diff omitted: …]` representation with conservative full-rewrite stats so a pathological patch cannot OOM the agent.
